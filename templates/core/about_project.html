2. Introduction and Objectives P.04

2.1 Introduction

The introduction provides an overview of the project, explaining the context, motivation, and significance of developing the system. It outlines the problem domain and describes how the project addresses real-world challenges or improves existing solutions.

The Net Profit Investment Tracking System (NPITS) is an advanced financial technology solution developed to bridge the gap between complex investment strategies and retail execution. In today's volatile equity markets, the difference between a successful investor and an amateur often lies in the precision of their data and the discipline of their rule-based approach. NPITS addresses the "Aggregation Paradox" — most brokerage platforms show a consolidated average price, which hides the profitability of individual stock "lots." For investors employing FIFO (First-In-First-Out) taxation or lot-based profit booking, this abstraction is a significant hurdle.

2.2 Objectives

The objectives define the goals that the project aims to achieve:

  1. Designing an Efficient, User-Friendly System: Implement a "Dashboard-First" interface that enables investors to view critical Buy Signals within seconds of logging in.
  2. Automating Manual Processes: Build robust connectors for CSV data and Google Sheets to ensure data parity without manual entry, automating the calculation of "Buy Gaps" and "Target Capitalization."
  3. Improving Data Accuracy & Security: Implement a strict FIFO engine that manages remaining_quantity at the transaction level with Decimal arithmetic for financial precision.
  4. Enhancing Accessibility & Scalability: Enable investors to separate their portfolio into 'Quant', 'Flexi', and 'Pyramid' strategies, each with its own health metrics.
  5. Demonstrating Practical Application: Apply theoretical knowledge of software engineering, database design, and financial computing to solve a real-world investment tracking challenge.

5. Problem Definition & Requirement Specifications P.20

5.1 Problem Definition

Most brokerage platforms show a consolidated average price, which hides the profitability of individual stock "lots." Investors employing FIFO taxation or lot-based profit booking face significant hurdles in tracking their true net profit per transaction. Additionally, there is no unified tool that advises investors on when to exit based on their specific average cost vs. target, combining real-time market data with rule-based strategy logic.

5.2 Functional Requirements

  • User Registration & Authentication: Custom registration with OTP-based email verification and social login (Google, GitHub, Apple).
  • Data Entry, Retrieval & Storage: Portfolio upload via CSV, Google Sheets sync, and manual entry with FIFO-based lot tracking.
  • Reporting & Analytics: Real-time P&L dashboards, strategy-wise distribution reports, and signal generation.
  • Notifications & Alerts: Buy/Sell/Reduce signal notifications displayed in the navbar with badge counts.
  • Administrative Control: Instrument verification, ticker symbol management, and user management via Django Admin.

5.3 Technical Specifications

  • Programming Language & Platform: Python / Django on Windows IIS / Linux Nginx
  • Database Design: Normalized relational schema with ACID-compliant transactions
  • API Integration: External LTP feeds for real-time price updates
  • Security Protocols: CSRF protection, parameterized queries, role-based access control, data encryption

9.3 Validation Rules (Applied on Data Structure)

To assure Data Quality and Integrity across all tables, strict validation rules are enforced at the Data Structure boundary:

  • Quantity Consistency: For any `Transaction`, `quantity > 0` and `remaining_quantity >= 0`. `remaining_quantity` can never exceed the original `quantity`.
  • Price Boundaries: Both `price` in Transactions and `avg_cost` in Portfolios must be `> 0.00`. Zero or negative prices are blocked at the ORM validation layer.
  • Duplicate Prevention (Composite Keys): A unique constraint `UNIQUE(user_id, instrument_id)` is applied on the `Portfolio` table. A user can only have one active aggregated portfolio row per stock.
  • Transaction Type Integrity: `transaction_type` is restricted strictly to an Enum of `['BUY', 'SELL']`. Any other string will raise a validation exception.
  • Orphan Record Prevention: Cascading deletes are controlled. If an `Instrument` is removed by an Admin, it is flagged as `is_active=False` rather than hard-deleted to preserve mathematical integrity in the User's `Transaction` history.

9.4 Data Normalization

The database schema is designed following the principles of database normalization up to the Third Normal Form (3NF) to minimize redundancy and prevent data anomalies.

  • First Normal Form (1NF): All table attributes contain atomic values. For example, the `USER` table separates `first_name` and `last_name` rather than a single combined name field, and the `PORTFOLIO` table uses a dedicated column for each distinct data point (quantity, avg_cost, etc.).
  • Second Normal Form (2NF): All tables meet 1NF, and all non-key attributes are fully functional dependent on the primary key. In the `TRANSACTION` table, details like `price` and `quantity` depend entirely on the specific transaction `id`, separated from the instrument details which reside in the `INSTRUMENT` table linked by a foreign key.
  • Third Normal Form (3NF): All tables meet 2NF, and there are no transitive functional dependencies. The system enforces this heavily: the `PORTFOLIO` table stores aggregated `avg_cost` and `total_quantity`, but the individual lot histories (which determine FIFO logic) are segregated into the `TRANSACTION` table. We derive Realized PnL dynamically rather than storing statically dependent values where possible, removing transitive dependencies.

9.5 Key Relationships & Constraints

  • Primary Keys: Auto-incrementing integer IDs for all core tables.
  • Foreign Keys: user_id references auth_user(id), instrument_id references core_instrument(id).
  • UNIQUE: UNIQUE(user, instrument) prevents duplicate stock entries for a single user.
  • CHECK: CHECK(quantity >= 0) ensures mathematical consistency in the portfolio.

10. System Structure & Modules P.55

10.1 Module Description

The system is decomposed into highly cohesive, loosely coupled modules:

  • Authentication Module: Custom implementation extending AbstractUser. Manages registration, OTP through SMTP, social login (Google, GitHub, Apple), and profile synchronization.
  • Portfolio Engine: The core calculator. Uses Decimal arithmetic for financial precision, avoiding floating-point errors. Implements the FIFO ledger for lot-based tracking.
  • Sync Controller: A specialized service that handles multi-source data ingestion (Google Sheets via URL, CSV Uploads) with data cleaning and validation.
  • Advisory Engine: Generates Buy/Sell/Reduce signals based on user-defined strategy rules and real-time LTP data.
  • Dashboard UI: A dynamic reporting layer using Chart.js for visualization and modern CSS for responsiveness.
  • Asset Class Extension: Specialized modules for Mutual Funds (SIP automation), Cryptocurrencies (24/7 price sync), and NPS (Retirement tracking).
  • Wealth Analytics: Captures daily snapshots of user wealth across all asset categories to generate long-term performance growth charts.

10.2 Procedural Logic (FIFO Engine)

When a user executes a SELL trade, the system follows this procedural flowchart:

1. Fetch all BUY transactions for User ID where remaining_qty > 0. 2. Sort by Date (ASC) and Created_At (ASC). 3. Loop through transactions: IF sell_qty <= tx.remaining_qty: tx.remaining_qty -= sell_qty Update Realized PnL Break ELSE: sell_qty -= tx.remaining_qty tx.remaining_qty = 0 Update Realized PnL Continue

10.3 Data Structures

Appropriate data structures used for each module:

  • Ordered QuerySets: For FIFO sorting of transaction lots by date and creation time.
  • Aggregation Dictionaries: For grouping portfolio data by instrument and strategy.
  • DataFrames (Pandas): For CSV and Google Sheets data ingestion and cleaning.

10.4 Implementation Methodology

NPITS was developed using the Iterative Enhancement Model combined with Agile/Scrum methodologies. Unlike a traditional waterfall approach, this allowed releasing a functional "Dashboard" version early and then layering on complex "FIFO" and "Advisory" logic in subsequent iterations.

12. Security Mechanisms P.80

Security is the cornerstone of any financial application. NPITS implements a multi-layered security architecture at multiple levels.

12.1 Authentication & Authorization

  • Custom User Model: Extended AbstractUser with OTP-based email verification.
  • Social Login: OAuth2-based authentication via Google, GitHub, and Apple.
  • Session Management: Secure cookie-based sessions with Django's built-in session framework.

12.2 Role-Based Access Control

  • Investor Role: Can view dashboards, upload portfolios, and manage their own profile.
  • Admin Role: Can verify instruments, manage system-wide ticker symbols, and administer users.

12.3 Data Encryption & Secure Communication

  • At-Rest Encryption: Production MySQL environment supports TDE (Transparent Data Encryption).
  • Application-level Field Encryption: Critical personal data (Mobile, DOB) and financial details (Loan amounts, Bank names) utilize `EncryptedCharField`, ensuring data is encrypted before it even reaches the database.
  • In-Transit Encryption: HTTPS enforced for all client-server communication.
  • Password Hashing: Django's PBKDF2 algorithm with SHA256 hashing for password storage.

12.4 Input Validation & Attack Prevention

  • Parameterized Queries: Django ORM prevents SQL Injection by design, using prepared statements.
  • CSRF Protection: Django's built-in CSRF middleware protects against cross-site request forgery.
  • XSS Prevention: Template auto-escaping prevents cross-site scripting attacks.
  • Referential Integrity: Foreign key constraints ensure data consistency and prevent orphan records.

18. Advanced Features & Automated Workflows P.105

NPITS v1.3.0 introduces several advanced features designed to enhance data integrity, user alerting, and system maintenance.

18.1 Financial Year Management & Locking

To ensure the historical accuracy of P&L data, the system now implements a robust Financial Year Tracking mechanism:

  • Data Persistence: Automated calculation and storage of invested value, unrealized PnL, and realized profits for each financial year.
  • Locking Mechanism: Financial years can be "locked" to prevent accidental modifications to historical data. Once locked, records are visually distinct (shaded) and edit-protected.
  • Chronological Data Integrity: Deletion of records is only permitted in reverse-chronological order (latest year first), ensuring that historical tax-critical data remains consistent.

18.2 Enhanced Signal Notification System

The updated signal engine provides a more intelligence-driven notification experience:

  • Signal Hashing: The system calculates a unique hash based on the current set of Buy/Sell/Reduce signals. This allows the system to detect if the *content* of the signals has changed, not just the count.
  • Persistent Badge Counts: Dynamic badge counts in the navigation bar alert users even before they click into the dashboard.
  • Signal State Management: The SignalNotificationState model tracks when a user was last notified, reducing redundant alerts.

18.3 Automated System Maintenance

For enhanced reliability and version control, NPITS now includes automated administrative workflows:

  • PowerShell Backup (Scheduled): A comprehensive backup script (create_backup.ps1) creates timestamped ZIP archives of the entire project, including the SQLite database and environment configurations, while excluding large log files and temporary caches.
  • GitHub Version Synchronization: Automated workflows to stage, commit, and push the latest verified version to GitHub, ensuring that the production codebase is always synced with the latest enhancements.

20. Extended Asset Class Modules P.112

NPITS has evolved beyond simple equity tracking into a holistic wealth management platform.

20.1 Mutual Funds & SIP Automation

The Mutual Fund module uses real-time NAV syncing from specialized financial data providers. The **SIP Engine** runs as a daily background task, automatically checking the `next_execution_date` for each active SIP and generating 'BUY' transactions without user intervention.

20.2 Fixed Assets & RD Compounding

Unlike regular assets, Fixed Assets (FD/RD) implement a specialized **Interest Compounding Engine**. It calculates the current value dynamically using the formula: A = P(1 + r/n)^(nt), adapted for monthly RD installments. The system also supports maturity tracking to ensure assets aren't over-counted beyond their term.

20.3 Multi-Liability Loan Tracker

The Loan module tracks liabilities using both **Flat** and **Reducing** interest methodologies. It provides a granular breakdown of each EMI payment into Principal and Interest components, giving users clear visibility into their actual depth of debt and remaining tenure.

20.4 Unified Net-Worth Dashboard

All assets and liabilities are aggregated into a single **Portfolio Dashboard**, which presents a real-time 'True Net Worth' by subtracting total liabilities (Loans) from total assets (Equity, MF, NPS, Crypto, Fixed, Other). This provides the ultimate financial health metric for the user.

20.4.1 Asset ID & Holder Tracking

With the expansion in v1.7.2, every Fixed Deposit and 'Other Asset' now supports granular **Holder Name** and **Asset ID** tracking. This ensures that assets spread across different banks or family members can be identified individually, facilitating smoother reconciliation and renewal management.

20.5 Cryptocurrency Dashboard & Global Price Sync

The Cryptocurrency module provides real-time tracking of digital assets. Unlike traditional markets, crypto operates 24/7. NPITS uses a specialized global price synchronization engine that pulls latest BTC, ETH, and other coin valuations from high-reliability financial data sources. It implements high-precision Decimal(20, 8) arithmetic to handle fractional units common in crypto transactions and calculates both realized and unrealized P&L using the standard FIFO methodology.

20.6 National Pension System (NPS) Tier-I/II Tracking

NPITS allows users to track their retirement corpus through the NPS module. It supports unit-based accounting for various Pension Fund Managers (PFMs). The system calculates the current valuation based on the latest available NAVs and provides a clear breakdown of Tier-I and Tier-II contributions, maturity projections, and historical growth.

20.7 IPO Activity Tracker

The **IPO Module** (formerly Donation) is being re-engineered to track Initial Public Offerings. While currently acting as a strategic placeholder in the navigation bar, its future roadmap includes automated subscription alerts, listing-gain analytics, and post-listing performance tracking for new entries in the Indian market.

20.8 Dividend Tracking & Yield Analytics

The **Dividend Module** allows users to log cash inflows from their equity holdings. It provides automated calculations for **Portfolio Yield** (Dividend as a percentage of Invested Value) and tracking of cumulative passive income per instrument.

20.9 Goal-Based Investment Planning

NPITS enables users to define specific **Financial Goals** (e.g., Retirement, Education, Property). The system maps current asset valuation against these targets, provided real-time progress indicators and estimated time-to-completion based on current contribution rates.

21. How it Works: The User Journey P.115

Understanding the flow of data within NPITS helps users maximize the system's rule-based advisory potential.

  1. Registration & Profile: User registers through OTP verification and optionally links social accounts. They define their "Investor Type" (Conservative to Aggressive) which sets the baseline for the Strategy Engine.
  2. Data Ingestion: Users can manually add transactions or use the Bulk Upload feature for Stocks and Mutual Funds. The Auto-Sync feature allows connecting a Google Sheet for seamless, zero-touch portfolio updates.
  3. FIFO Processing: As transactions are added, the backend sorts them chronologically. Sell orders are automatically mapped to the oldest "BUY" lots, calculating the exact realized profit for every share sold.
  4. Real-Time Sync: At regular intervals, the system's sync job checks for LTP/NAV updates for all instruments in the user's portfolio.
  5. Signal Generation: The Rule-Engine compares the LTP with the User's Average Cost and Strategy targets. If a stock hits a gain/loss threshold, a Signal is generated (e.g., "Buy at 30% gap", "Sell at 22% target").
  6. Actionable Insights: The user views their Dashboard, sees the Signal Badge count, and uses the "Buy/Sell" insights to rebalance their portfolio per their chosen strategy.
